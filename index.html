VLM-Guided Spatial Reasoning for Robotic Tool Use

Full ToolMind overview demo across all three constrained manipulation scenarios: wall retrieval, screwdriver insertion, and shelf retrieval.

We couple VLM semantic reasoning (GPT-5.5) with depth-based geometric refinement to drive multi-phase robotic tool use. ToolMind uses Structured Chain-of-Thought prompting to mitigate spatial myopia, while DBSCAN cluster labels and depth back-projection turn VLM outputs into actionable 3D keypoints.

Abstract

ToolMind is a dual-modality pipeline for scenario-specific sequential robotic tool use. Given RGB-D observations of a tabletop scene, the system identifies the correct tool, predicts task-relevant grasp, functional contact, and waypoint keypoints, and executes a multi-phase manipulation sequence. GPT-5.5 handles semantic scene understanding, tool selection, structured keypoint reasoning, and waypoint generation, while a depth-based geometric module refines coarse VLM outputs into precise 3D keypoints.

The central challenge is spatial myopia: VLMs often regress keypoints toward object centroids instead of the task-relevant extremities needed for tool use. We mitigate this with a Structured Chain-of-Thought JSON prompting strategy that forces geometry → grasp anchor → contact tip reasoning before coordinate regression. The resulting keypoints drive KOMO-based trajectory optimization in a simulated Franka Emika Panda environment across wall retrieval, screwdriver insertion, and shelf retrieval tasks.

The pipeline requires no task-specific VLM retraining and operates in an open-world style: DBSCAN-derived cluster labels and per-cluster geometry summaries anchor the model's attention, while depth back-projection, point-cloud snapping, PCA geometry, and antipodal search convert VLM predictions into physically feasible robot actions.

Key Contributions

Cluster-anchored VLM reasoning

DBSCAN clusters are projected back to RGB views as labelled regions, turning open image regression into a structured selection-and-localization problem.

Structured CoT against spatial myopia

The prompt requires explicit geometry, grasp-anchor, and contact-tip reasoning before GPT-5.5 is allowed to output coordinates.

Depth-based 3D refinement

VLM pixels are lifted through the registered depth channel, snapped to the selected tool point cloud, and refined with PCA and antipodal search.

Scenario-specific KOMO execution

The refined keypoints guide grasping, positioning, and execution phases for pull, insert/rotate, and sweep/push manipulation.

Method

Four tightly-coupled stages connect raw RGB-D perception to physics-based trajectory execution. GPT-5.5 acts as the semantic backbone — selecting tools, predicting keypoints, and generating motion planning waypoints.

Stage 01
📷
Scene Perception

RGB-D capture from top-down + oblique views. Depth back-projected to 3D. DBSCAN clustering, alphabetic object labeling.

Stage 02
🧠
VLM Reasoning

GPT-5.5 receives annotated RGB + cluster summaries. Outputs tool label, grasp region, contact pixel, and motion planning waypoints via CoT prompting.

Stage 03
📐
Geometric Refinement

Contact pixel back-projected to 3D via depth channel. PCA axis estimation, antipodal grasp search, vector-based tool orientation.

Stage 04
🤖
KOMO Execution

Multi-phase trajectory optimization: grasp → position → execute. Collision avoidance + joint-limit constraints on Franka Panda.

Depth Channel: 2D Pixel → 3D World

The VLM operates on RGB images and outputs pixel coordinates [u, v]. These are lifted to 3D world-frame keypoints using the depth channel via camera intrinsics.

The resulting camera-frame point is transformed to world coordinates using the camera extrinsics. The VLM-predicted functional contact pixel is back-projected this way and then snapped to the nearest point on the selected tool's point cloud, yielding the refined 3D contact keypoint c.

VLM's Role in Motion Planning

Beyond tool selection and keypoint prediction, GPT-5.5 serves as a semantic motion planning assistant — providing waypoints that guide the KOMO trajectory optimizer for each scenario.

🗺️ Waypoint Generation

The VLM is queried for approach angles, target positions, and directional vectors. These serve as semantic waypoints that constrain the KOMO optimizer's trajectory phases.

🔄 Multi-Query Re-Planning

The VLM is queried multiple times during execution — e.g., to determine a drag endpoint or an updated cup position after sweeping — enabling closed-loop semantic guidance.

📍 Orientation Estimation

For constrained spaces, the VLM identifies key geometric points (A, B for sweep direction; G, C for tool axis) that drive vector-based tool orientation alignment.

🧩 Semantic Scene Understanding

VLM analyzes cluster positions, object identities, and spatial relationships to reason about which side to approach from and how far objects must be moved to become reachable.

Structured Chain-of-Thought Prompting

To counter spatial myopia, the JSON schema mandates three explicit reasoning steps before any coordinate regression is allowed:

1
step1_geometry_analysis

Describe the physical shape of the selected tool and locate its extreme boundaries in the image.

2
step2_grasp_location

Identify the exact region designated for robotic contact — e.g., "the right-most end of the stick".

3
step3_contact_tip_location

Using the grasp location as anchor, deduce the exact opposite functional extremity. Only then is the pixel coordinate [u, v] requested.

Prompt Levels (P0 – P3)

P0 — Minimal

Single instruction: select a tool and output pixel coordinates. No schema, no task context.

P1 — Structured

Defined JSON schema with named fields for tool, grasp region, pixel coordinates, and confidence. No scenario context.

P2 — Scenario-Aware

P1 plus full scene context and, where applicable, calibrated handle coordinate hints from gt.json.

P3 — Anti-Failure BEST

P2 plus explicit anti-failure constraints such as "do not grasp the functional tip" and self-check reasoning fields.

VLM Contributions per Scenario

🧱

Scenario 1: Wall Retrieval (Pulling)

PULL · Tip Localization

A target object is placed behind a rigid wall, beyond the robot's direct reach. The system selects a long, rigid tool, grasps it from one end, and uses the opposite tip to drag the object into the reachable workspace. The primary geometric challenge is accurate functional tip localization: the contact point must be at the far end of the tool, not its center.

VLM's role in motion planning

During the motion planning stage, the VLM semantically understands the environment and performs semantic planning over the detected clusters. It is queried to determine the approach angle and the direction in which the functional tip should be oriented — acting as a waypoint generator. The functional tip is then aligned toward this VLM-defined point. Afterwards, the VLM is queried a second time to determine how far the target must be dragged so that it becomes reachable by the robot, providing a drag endpoint waypoint that terminates the pull motion.

🔩

Scenario 2: Screwdriver Insertion

INSERT / ROTATE · Non-translational Planning

A target object requires actuation via a rotational motion. The VLM selects a screwdriver-like tool from the segmented point cloud clusters, and the best antipodal grasp point is computed within the VLM-specified handle region.

VLM's role in motion planning

The VLM analyzes the scene and determines an appropriate rotation waypoint for the screwing motion. This waypoint is also used to align the tool along the vertical axis before the screw motion begins. Once the functional tip (back-projected to 3D from the VLM's contact pixel via the depth channel) reaches the target position, the KOMO planner executes the rotational trajectory. The VLM thus guides a non-translational end-effector motion entirely through semantic scene reasoning.

📦

Scenario 3: Shelf Retrieval (Sweeping)

SWEEP → GRASP · Constrained Insertion

A cup placed under a low-clearance shelf cannot initially be reached by the robot. The robot must navigate a pushing tool through the narrow gap and sweep the cup into a position within the robot's reachable workspace. This is the most geometrically constrained scenario: incorrect tool orientation causes shelf collisions or joint-limit violations.

VLM's role in motion planning

The VLM is queried in a Chain-of-Thought manner to identify the functional tip (C) and grasp point (G), constructing the tool axis vector GC. It also determines two points A and B behind the shelf edges — one near the robot side, one at the far side — defining the sweep direction vector AB. The tool orientation is then adjusted until AB ⊥ GC (computed via dot product), preventing shelf collisions. The robot sweeps with the tool guided by the VLM-defined point A and height. After sweeping, the cup is brought into a position within the robot's reachable workspace. This multi-query VLM strategy makes the pipeline robust to domain randomization and varying initial tool placements.

Shelf Rollout Analysis

With Structured CoT

In the shelf scenario, 6 out of 10 rollouts are successful. Among the 4 failures, 2 are caused by selecting the wrong A–B shelf-edge pair, while the other 2 occur because the predicted C point is placed on the same tool end as the grasp instead of the opposite functional tip.

Before CoT

Before adding Chain-of-Thought reasoning, only 3 out of 10 shelf rollouts were successful. The main failure mode was spatial myopia: C was often predicted near the grasp region instead of the distal contact tip. In those failures, 2 came from the wrong A–B edge pair and 5 came from incorrect C-point selection.

Key Findings

Structured CoT targets functional contact localization

The main shelf gain comes from reducing C-placement failures from 5 to 2; the A–B edge-pair error stays at 2 failures in both settings.

Cluster labels reduce ungrounded coordinate regression

DBSCAN-derived labels and cluster summaries anchor the VLM to candidate tool regions, converting raw coordinate regression into selection-then-localization.

Shelf retrieval is the hardest scenario

The shelf task requires relational geometry over A, B, grasp G, and functional contact C, making it more sensitive to spatial myopia than wall or screwdriver tasks.

Execution remains mostly open-loop

Once the KOMO trajectory is generated, the system does not yet re-plan after unexpected slip or contact; closed-loop recovery is a natural next step.
